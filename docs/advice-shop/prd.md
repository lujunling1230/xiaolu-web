## 1. 产品背景与定位

### 1.1 产品背景

在快节奏的现代生活中，人们常常面临情感困惑却无处诉说。传统的心理咨询门槛高、成本大，而社交媒体上的倾诉又缺乏深度和安全感。用户需要一个可以随时倾诉、无需顾虑评判、能获得温暖回应的空间。

解忧杂货铺 (The Advice Shop) 的灵感来源于东野圭吾同名小说中的"解忧杂货店"概念 -- 一个可以写信倾诉烦恼并收到回信的神秘小店。产品将这一文学意象转化为数字体验，借助大语言模型（阿里云百炼 DeepSeek），为用户提供一位始终在线、温和有趣的"晴美"掌柜，让每一次倾诉都能获得真诚的回应。

### 1.2 产品定位

解忧杂货铺是「妙妙工具箱」中"术"维度的代表性工具，定位于**效率与工具**。它替代了原"万能百事通"工具，以更聚焦、更情感化的方式提供 AI 问答服务。在全站 9 维度体系中，"术"维度关注的是借助工具提升生活效率和质量的能力。

| 维度 | 产品定位 | 关键指标 |
|------|----------|----------|
| 术 (Technique) | AI 情感问答空间 | 回信生成率、用户留存率 |
| 风格 | 暖色调信纸风 | 米白底 + 金色边框 + Noto Serif SC 衬线字体 |
| 用户心智 | 温暖树洞 | "总有一句话，能解开你的心结" |

### 1.3 产品目标

- **情感陪伴**: 提供一个无压力的倾诉空间，让用户获得被倾听的感觉
- **即时反馈**: 通过伪流式打字机效果，营造"正在写信"的真实感
- **零门槛体验**: 无需登录注册，打开即用，输入问题即可获得回信
- **角色一致性**: 固定"晴美"角色，提供稳定、可预期的人格化交互体验

## 2. 用户画像与核心场景

### 2.1 用户画像

| 画像标签 | 描述 | 典型行为 |
|----------|------|----------|
| 情绪低谷者 | 20-35 岁，工作压力大，偶尔感到迷茫 | 深夜打开解忧杂货铺，倾诉职场困惑或人生方向 |
| 孤独感寻求者 | 18-30 岁，独居或社交圈较小 | 把晴美当作日常聊天的朋友，分享琐事和心情 |
| 好奇心探索者 | 任何年龄段，对 AI 对话感兴趣 | 测试各种问题，观察 AI 回信的风格和趣味性 |
| 文学爱好者 | 25-40 岁，喜欢东野圭吾作品 | 被"解忧杂货铺"概念吸引，体验数字版"写信-回信" |

### 2.2 核心场景

**场景一：深夜情绪倾诉**

> 用户深夜加班回到家，感到疲惫和孤独。打开解忧杂货铺，在信纸界面写下"工作好累，不知道这样拼下去有什么意义"。几秒后，屏幕上字符逐个浮现，晴美用轻松幽默的语气回应，让用户感到被理解。

**场景二：日常闲聊解闷**

> 用户无聊时打开解忧杂货铺，随意输入"今天下雨了，心情也跟着阴沉沉的"。晴美用俏皮的口吻回应，聊天气、聊雨天适合做的事，像朋友一样自然地接话。

**场景三：决策困惑咨询**

> 用户面临两个工作机会的选择，输入"两家公司都给了 offer，一家钱多但加班多，一家钱少但轻松，怎么选？"。晴美不给出绝对建议，而是用生活化的方式帮用户梳理自己的真实想法。

**场景四：连续对话探索**

> 用户对晴美的回信感到有趣，点击"再写一封"开启新一轮对话，测试不同话题下晴美的反应，体验 AI 角色扮演的趣味性。

### 3.2 模块详情

#### 3.2.1 信纸界面模块

**功能描述**: 页面核心交互区，以信纸风格呈现问题输入表单。

**界面元素**:

| 元素 | 说明 |
|------|------|
| 温馨开场白 | 随机三选一："夜安，旅人。" / "嘿，我在听。" / "别急，慢慢说。" |
| 输入框 | 4 行 textarea，信纸横线背景 |
| 占位提示 | "写下你的疑惑...（比如：为什么努力了还是没结果？）" |
| 投递按钮 | 全宽渐变金按钮，文字"投递信件" |

**交互规则**:
- 输入框为空时，投递按钮为禁用状态（50% 透明度）
- 加载中时，投递按钮文字变为"正在写信..."，禁用状态
- 支持 Ctrl+Enter (Windows) 或 Cmd+Enter (Mac) 快捷键提交
- 开场白基于 `question.length % 3` 循环切换，保持稳定不闪烁

#### 3.2.2 AI 回信模块

**功能描述**: 调用阿里云百炼 DeepSeek 模型，生成晴美角色的回信，以伪流式打字机效果逐字展示。

**角色设定 -- 晴美 (Harumi)**:

| 维度 | 设定 |
|------|------|
| 身份 | 解忧杂货店的看店掌柜，普通但有趣的女生 |
| 风格 | 松弛感、自然接话、幽默风趣、简短精炼 |
| 语言 | 口语化，允许使用"呗""呀""哎""害"等语气词 |
| 长度 | 一句话能说清的不说两句，避免长篇大论 |

**系统提示词设计要点**:

```
你是晴美，一个普通但有趣的女生。你在解忧杂货店里看店，闲着没事就找人唠嗑。
这里是聊天室，不是邮局。用户说什么，你就聊什么，别强行把话题拉回什么"信件"、"邮票"、"墨水"之类的。

【核心风格】
- 松弛感：像朋友一样闲聊。可以自嘲、可以八卦、可以跑题。
- 接话自然：不查户口，不质问，顺着用户的话头聊下去。
- 幽默温度：保持幽默感，回复要短、快、有趣。
- 简短精炼：一句话能说清的，绝不说两句。

【绝对禁止】
- 严禁提及"写信"、"贴邮票"、"等信"、"墨水"、"投递"、"信封"等与写信流程相关的词汇。
- 严禁在正文末尾添加任何署名、落款或破折号。
- 严禁使用括号进行动作描写。
- 严禁使用"建议"、"希望"、"首先"、"相信"等AI或客服词汇。
- 严禁承认自己是AI或模型。
- 严禁解释回复动机。
```

**伪流式打字机效果实现**:

```
const fullText = data.choices?.[0]?.message?.content || "";
setReply("");
for (let i = 0; i < fullText.length; i++) {
  await new Promise((resolve) => setTimeout(resolve, 30));
  setReply(fullText.slice(0, i + 1));
}
```

- 每字符间隔 30ms
- 逐字更新 `reply` 状态
- 使用 `await` + `setTimeout` 实现异步延迟

**回信卡片展示**:

| 元素 | 内容 |
|------|------|
| 印章 | "已送达"（右上角，旋转 8 度） |
| 称呼 | "亲爱的旅人：" |
| 角色署名 | "晴美 回信" |
| 正文 | AI 生成的回信内容 |
| 签名 | "—— 晴美 · 解忧杂货店" |
| 操作 | "再写一封"按钮 |

#### 3.2.3 加载与错误处理模块

**加载状态**:
- 显示钢笔图标（摇摆动画）+ "杂货店老板正在回信..." 文案
- 钢笔图标左右摇摆（rotate: [-3, 3, -3]，1.2 秒循环）

**降级处理**:
- 当环境变量未配置（无 API Key 或 Base URL）时，不发送请求
- 600ms 后展示默认回信："网有点卡，稍等一会儿呗。"

**错误处理**:
- 网络错误：展示"信还在路上...\n\n（调试信息：具体错误信息）"
- 响应为空：抛出"大模型返回了空内容，请检查模型配置"
- 请求取消：静默处理（AbortError），不展示任何错误信息

**请求取消**:
- 每次新请求前，调用 `abortRef.current?.abort()` 取消上一次未完成的请求
- 使用 `AbortController` + `signal` 实现

## 4. 数据模型

### 4.1 无持久化存储

解忧杂货铺不进行任何数据持久化。所有对话仅存在于当前会话中，页面刷新或关闭后数据完全消失。这一设计决策基于以下考量：

- **隐私优先**: 用户倾诉的内容可能涉及个人隐私，不存储数据是最彻底的保护方式
- **轻量体验**: 无需管理数据生命周期，降低技术复杂度
- **场景匹配**: 情感倾诉更适合"一次性"的体验，不留下痕迹反而增加安全感

### 4.2 组件状态

```typescript
// 组件内部状态
const [question, setQuestion] = useState("");       // 用户输入的问题
const [reply, setReply] = useState<string | null>(null);  // AI 回信内容
const [loading, setLoading] = useState(false);      // 加载状态
const [currentRole, setCurrentRole] = useState<string | null>(null);  // 当前角色
const abortRef = useRef<AbortController | null>(null);  // 请求取消控制器
```

### 4.3 环境变量配置

## 5. AI 调用策略

### 5.1 模型选择

| 属性 | 说明 |
|------|------|
| 模型 | 阿里云百炼 DeepSeek（deepseek-r1） |
| 调用方式 | OpenAI 兼容 API 格式 |
| 请求方式 | POST，非流式（stream: false） |
| 角色设定 | 系统提示词注入（role: "system"） |

### 5.2 请求参数

```json
{
  "model": "deepseek-r1",
  "messages": [
    {
      "role": "system",
      "content": "你是晴美，一个普通但有趣的女生..."
    },
    {
      "role": "user",
      "content": "用户输入的问题"
    }
  ],
  "stream": false
}
```

### 5.3 响应解析

```typescript
const data = await response.json();
const fullText = data.choices?.[0]?.message?.content || "";
```

### 5.4 策略选择说明

- **非流式 + 伪流式**: 采用非流式 API 请求获取完整回信，再通过前端逐字打印模拟流式效果。这一策略相比真正的 SSE 流式请求，具有更好的兼容性和可控性，且回信内容完整后再展示，避免中途断连导致的残缺回复。
- **固定角色**: 角色固定为"晴美"，不提供角色切换功能。这一决策简化了用户体验，避免选择焦虑，同时确保角色人格的深度打磨。
- **无上下文记忆**: 每次请求独立，不携带历史对话。这降低了 token 消耗，同时避免了长对话中角色人格漂移的问题。

## 6. 交互逻辑与规则约束

### 6.1 提交校验

- 问题文本去除首尾空格后非空才可提交
- 加载中时禁止提交（按钮禁用 + 视觉反馈）
- 支持 Ctrl+Enter / Cmd+Enter 快捷键提交

### 6.2 请求管理

- 每次新请求前自动取消上一次未完成的请求
- 使用 AbortController 管理请求生命周期
- 取消请求时静默处理，不展示错误提示

### 6.3 回信展示

- 回信使用伪流式打字机效果逐字打印
- 打印完成后展示完整回信卡片
- 回信卡片使用 AnimatePresence 实现入场/退场动画
- 加载状态和回信状态互斥（同一时间只展示一个）

### 6.4 重置逻辑

- 点击"再写一封"清空回信和输入框
- 回到初始信纸状态，等待新一轮输入
- 开场白随 question.length 变化而更新

### 6.5 路由配置

| 路由 | 页面 | 说明 |

### 6.6 响应式设计

- 最大内容宽度 680px，居中显示
- 移动端（<=520px）信纸和回信卡片内边距缩减至 28px 22px

## 8. 未来迭代方向

### 8.1 V1.1 短期迭代

- **角色扩展**: 新增 2-3 个可选角色（如"温柔的奶奶""毒舌的朋友"），丰富对话体验
- **对话历史**: 当前会话内保留对话历史，支持回看之前的问答
- **一键复制**: 回信卡片增加复制按钮，方便用户保存温暖的回信
- **主题切换**: 新增深色模式（深夜信纸风格）

### 8.2 V1.2 中期规划

- **上下文记忆**: 在同一会话内携带最近 N 轮对话，提升回信连贯性
- **情感分析**: 识别用户情绪倾向，调整回信语气和风格
- **语音输入**: 支持语音转文字输入问题
- **回信朗读**: TTS 朗读回信内容，增加沉浸感
- **分享卡片**: 生成精美的回信分享图片

### 8.3 V2.0 远期愿景

- **多模态支持**: 用户可上传图片（如手写信），AI 识别并回信
- **真实信件**: 支持用户填写真实地址，定期收到晴美寄出的实体明信片
- **社区广场**: 匿名分享回信（经用户同意），形成温暖社区
- **个性化角色**: 用户可自定义角色人设和语气
